# PQ-AI 开发计划

## 1. 业务目标

PQ-AI 以“生产事件 + 测量点”为主线，将中涂、色漆、清漆三个涂层体系中的五个喷涂执行阶段、Dürr 机器人/旋杯/轨迹、材料实绩和漆膜质量结果关联起来，形成：

1. 质量预测：在质量测量前预测膜厚、橘皮、色差/效应风险。
2. 问题诊断：在排除仪器与数据质量问题后，定位对异常点位影响最大的工艺、材料和设备相关因素。
3. 参数推荐：在设备硬边界、材料 TDS 和工艺规则约束内生成调整建议。
4. 受控闭环：推荐经人工审批后生成程序版本，执行、复测并评价真实改善效果。

首版不允许 AI 直接修改机器人参数；所有推荐必须可解释、可审批、可追溯、可撤销。

明确排除预处理、电泳、涂胶、喷房温湿度、烘房温度、调漆间和光泽度。现有演示数据与目录中的越界字段必须先隔离，再重建特征和模型。

## 2. 领域划分

### 主数据

- 工厂、车型、颜色、零件、测量编组、测量点。
- 颜色资料包含供应商、特征值、TDS、MSDS、COA、DOE 和数字标准。
- 点位必须关联零件、区域、质量指标类型和匹配点属性。

### 工艺程序

- 喷涂程序支持多个车型和多个颜色。
- 程序按版本管理，版本包含多个刷子表和刷子号参数。
- 刷子号与测量点通过贡献权重关联，用于处理喷幅重叠。
- 参数采用“参数定义 + 参数值”建模，覆盖五个执行阶段的流量、转速、内外成型空气、静电、距离、间距、速度等。
- 机器人、旋杯、控制器、钟罩、轨迹程序/校验和、TCP、路径段、姿态、触发和重叠必须版本化。

### 生产实绩

- 生产事件记录工厂、车型、颜色、批次、班次和时间。
- 每个生产事件包含五个喷涂执行阶段实绩、实际程序版本、材料批次和实际参数。
- 配置参数与生产实绩分开保存，AI 训练优先使用实际参数。

### 质量与标准

- 质量测量统一记录测量编号、编组、点位、数据类型、状态、时间和测量人。
- 橘皮、色差/效应和膜厚指标以标准化指标值存储，并关联适用标准。
- 测量必须关联仪器、探头/方法、固件/导入模板、校准/参考状态、重复测量和原始数据。
- 标准支持工厂、车型、颜色、零件、点位、数据类型和有效期等适用范围。

### AI 与闭环

- 点位特征快照保存一次预测所使用的完整工艺、材料和上下文特征。
- 模型版本记录训练数据血缘、指标、制品地址、上线状态和适用范围。
- 预测、诊断、推荐、审批、执行和复测评价全程留痕。
- 数据集按生产事件/车身/批次分组并按时间验证，禁止同一事件点位跨训练/验证集。
- 候选模型必须生成时间留出、生产分组留一和可用工厂/车型/颜色验证轴证据；缺少轴多样性必须明示，不得伪造跨域验证。
- 模型工件必须登记哈希，验收和激活时复核哈希一致性。
- 推荐审批前必须创建并批准受控试验计划，记录假设、证据类型、风险、回滚和持续观察方案。
- 解释只代表关联；推荐必须基于约束并经受控试验验证。

## 3. 核心数据链

```text
生产事件
  -> 三个涂层体系 / 五个执行阶段实绩
  -> Dürr 设备、程序、轨迹与实际参数
  -> 按目标族版本化的点位贡献
  -> 受治理的材料批次特性
  -> 点位特征快照
  -> 有仪器/方法/校准血缘的质量测量与标准判定
  -> 预测 / 诊断 / 推荐
  -> 人工审批 / 新程序版本
  -> 执行 / 复测 / 效果评价
```

## 4. 实施阶段

### 阶段 0：工程基础与产品原型

当前已完成：

- Next.js 管理端、FastAPI 服务和 MySQL Docker 编排。
- 核心领域表的 SQLAlchemy 模型；MySQL 结构变更不由代码自动执行，必须走审批工单并由人工执行 SQL。
- 已按公司 MySQL 规范建立基线：无物理外键，应用层逻辑外键，禁止运行时物理 `DELETE` 和自动 DDL，生成 DBA 审批建表 SQL。
- 工艺质量驾驶舱、程序中心、质量中心、AI 工作台和主数据中心界面。
- Dashboard、主数据、程序、质量、点位特征、预测、诊断、推荐和审批 API。
- 驾驶舱实时 API 数据读取、降级快照和真实审批 API 调用。
- 42 个工艺/材料参数定义目录、62 个受治理质量指标目录和幂等演示数据。
- 基于刷子/轨迹贡献权重、实际参数、生产前已验证材料批次结果和质量标签的点位特征快照。
- 可持久化岭回归基础模型训练、真实预测、相关性诊断和硬边界约束推荐。
- 推荐审批、执行值记录、复测质量关联和闭环效果评价。
- 用户名/密码登录、HttpOnly 会话 Cookie、API Key 系统集成认证、标准角色权限目录、关键写操作审计与审计中心。

### 阶段 1：范围清理与领域模型整改

- 已完成：删除或隔离目录、演示数据、导入、快照与模型中的越界字段。
- 明确三个涂层体系与五个执行阶段，迁移历史命名。
- 已完成仪器/方法/校准/参考/导入模板与重复读数、Dürr 设备/轨迹和受治理材料特性基线；继续增加独立探头/MSA、真实设备文件和真实 TDS/COA 适配。
- 已完成 3C3B 工艺路线、路线步骤、适用车型/颜色、质量问题/调试工单、证据、协作记录和诊断知识库基线；工程师可从异常进入可审计任务流并关联生产、点位、材料、测量和推荐上下文。
- 已完成通用设备/材料文件导入 profile、base64 文件预览、字段映射校验、错误报告、重放审计与导入 job 治理；导入预览不自动写目标业务表，不触发 DDL。
- 已完成测量探头、MSA/GRR、供应商材料提交/问题、点位贡献验证、轨迹段几何和模型解释基线；下一步扩展真实 BYK/Fischer/Dürr/COA/TDS 文件解析器和字段级设备向导。
- 验收：越界字段无法进入正式快照和模型；任一工艺和测量事实具备完整血缘。

当前范围策略已覆盖 API、集成事件、种子数据、特征快照、模型激活/推理、前端和测试；旧版模型保留血缘并退役，合规演示模型已重训。仪器可靠性与测量追溯、显式探头/MSA、Dürr 设备配置/轨迹校验和/生产执行、目标族贡献、贡献验证、轨迹几何、材料治理、供应商材料提交、v4 点位血缘、无泄漏模型验收、多轴验证折、模型工件登记、模型解释、工厂/车型/颜色适用范围、OOD 阻断、工厂验收策略版本、受控试验计划门禁、工程问题工单、约束来源版本和回滚执行记录基线已完成，下一项为真实设备/材料文件适配、不确定度、自动程序回滚发布和持续稳定性看板。

### 阶段 2：主数据与程序版本管理

- 完成工厂、车型、颜色、零件、编组、点位、程序、刷子表的完整 CRUD。
- 所有有领域 CRUD 的页面必须提供 Excel/CSV 模板、批量导入和批量导出；导入前通过模板固定字段名、数据类型、必填项和业务键，导入时执行应用层引用检查与 upsert。
- CRUD 中的删除语义改为停用、归档、状态流转或版本替换；正式 HTTP `DELETE` 保持禁止。
- 完善字段级权限、版本差异、封样和审批流程。
- 在实际 MySQL 测试环境执行审批 SQL、回滚 SQL 和人工执行记录验证。
- 验收：可从界面建立一套可追溯的工厂、车型、颜色、程序和点位关系。

### 阶段 3：质量数据与点位聚合

- 接入 MES、机器人、材料和测量设备数据，建立可重放的数据导入任务。
- 批量导入只写入既有表和既有字段，不允许通过文件内容触发 `CREATE`、`ALTER`、`DROP`、`SET` 或物理 `DELETE`；生产环境新增/变更字段仍按 DBA 审批 SQL 人工执行。
- 实现刷子贡献权重维护和点位特征聚合服务。
- 实现质量标准判定、异常标记、缺失值和数据质量监控。
- 验收：任一质量测量结果均可追溯到生产事件、程序版本、材料批次和点位特征。

### 阶段 4：质量分析与规则诊断

- 上线 SPC、趋势、标准符合性、车型颜色对比和点位热力图。
- 建立基于工艺知识库的规则诊断和硬边界检查。
- 验收：异常点位可获得可解释的规则诊断及关联证据。

当前已完成通用质量标准自动匹配、上下限判定、超差原因输出、SPC 控制图、趋势斜率、Cp/Cpk、点位风险热力图和基础数据质量监控；BYK/Fischer 仪器、方法、参考件、校准、导入模板、重复读数及 `VERIFIED` 可靠性门禁已接入，真实设备文件适配、MSA 和目标族诊断规则待开发。

### 阶段 5：AI 模型与 MLOps

- 建立训练数据快照、特征版本、模型注册、离线评估和在线监控。
- 分目标族训练预测模型；以 SHAP、敏感性分析和约束优化支持关联诊断与推荐。
- 建立按生产事件/车身/批次分组并按时间留出的离线评估与验收门禁。
- 设置车型、颜色、工厂适用范围及模型降级策略。
- 验收：模型达到约定精度，且每次结果均可追溯到模型版本和特征快照。

当前已完成不可变数据集快照、目标测量血缘、按车身/生产运行分组的时间留出、泄漏检查、仅训练集拟合、独立验证指标、多轴验证折、模型工件哈希登记、人工验收和激活门禁，以及岭回归基线、预测、相关性诊断和在线漂移监控。模型训练会派生待验收的工厂/车型/颜色适用范围和统计 OOD 阻断策略；人工验收要求全部适用工厂存在生效的目标指标验收策略，并满足验证 RMSE、验证 R²、独立分组数、已评估验证轴和工件哈希一致性。范围外、缺失和分布外输入会阻断预测与推荐并保存治理证据。真实工厂策略配置、外部模型制品仓库、丰富不确定度、自动降级与更强模型族仍待开发。

### 阶段 6：受控工艺闭环

- 推荐通过硬安全边界、TDS、单次调整步长和冲突校验。
- 审批后生成新程序版本并进入受控试验，不直接覆盖当前生效版本。
- 程序版本、关联关系和质量/轨迹明细更新不得采用删除重建；必须使用版本化、追加、upsert 或审批后的状态变更。
- 执行后自动关联复测结果，评价实际改善、偏差和副作用。
- 验收：完成“发现风险 -> 推荐 -> 审批 -> 执行 -> 复测 -> 评价”的审计闭环。

当前已完成硬边界约束推荐、审批、执行、复测和效果评价 API 及操作界面；MES、QMS、机器人和材料系统的幂等事件、失败重试、死信和重放框架已实现，材料事件可写入受控批次检测结果并自动判定可靠性，真实现场地址、凭证与字段映射待配置。

当前已补充受控试验计划门禁和回滚记录：推荐审批前必须创建并批准包含假设、证据类型、预期、风险、回滚和持续观察方案的 `controlled_trial`；推荐动作必须固化 `parameter_constraint_source` 约束来源版本；执行推荐会自动把试验推进到 `RUNNING`，复测评价会把试验归档为 `VERIFIED` 或 `INEFFECTIVE`，失败试验可通过 `program_rollback_execution` 记录回滚动作快照。下一步需要把回滚记录接入真实程序版本发布、PLC/MES 执行和持续稳定性看板。

### 阶段 7：生产部署与持续运营

- 对接企业 SSO，并完成备份、灾备、告警和可观测性。
- 前端部署至 Vercel 或企业 Web 平台，FastAPI、AI Worker 和 MySQL 部署在企业内网。
- 以工厂为单位灰度上线，持续监控模型漂移和闭环收益。

当前已完成用户密码登录、服务端会话、API Key 鉴权、RBAC 角色权限目录、写操作审计和审计中心；企业 SSO、
密钥托管与定期轮换仍需结合客户基础设施落地。

## 5. 关键验收指标

- 数据关联完整率、点位特征可用率和实际参数覆盖率。
- 各质量指标预测误差、风险召回率和误报率。
- 推荐通过率、审批率、真实改善率和平均闭环时长。
- 所有程序变更、模型结果和审批操作的审计可追溯率为 100%。
- MySQL 运行时 DDL、物理删除、物理外键和未经审批的结构变更为 0。
