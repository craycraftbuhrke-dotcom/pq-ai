# Sensitive Configuration Inventory

This project must not ship real or example secrets in source code, Dockerfiles,
Compose files, frontend bundles, tests, or documentation.

| Name | Injection time | Owner | Sensitive | Browser visible | Rule |

| `MYSQL_PASSWORD` | MySQL container runtime | Kubernetes Secret / Compose Secret / local ignored `.env` | Yes | No | Required only when local Compose manages MySQL. |
| `MYSQL_ROOT_PASSWORD` | MySQL container runtime | Kubernetes Secret / Compose Secret / local ignored `.env` | Yes | No | Required only when local Compose manages MySQL. |
| `CI_REGISTRY_PASSWORD` | CI runtime | CI/CD masked protected variable | Yes | No | Used only by CI jobs when pushing images. |
| `PIP_CONFIG_SECRET_FILE` | Backend image build | GitLab masked/protected File variable + BuildKit secret | Yes | No | Mounted only for pip install; must not be passed as build-arg or persisted in image history/layers. |
| `KUBE_CONTEXT` | GitLab deploy job | CI protected variable / GitLab Kubernetes Agent | Yes | No | Grants access to the target cluster context. |
| `pq-ai-runtime-secrets/database-url` | Kubernetes backend runtime | Kubernetes Secret managed by platform | Yes | No | Referenced by name only in the manifest. |
| User passwords | API request runtime | User input / enterprise identity provider | Yes | No | Never prefilled or stored in source. Stored only as salted password hashes in MySQL. |
| API keys issued by the app | API request runtime | Security admin workflow / integration owner | Yes | No | Generated by the backend using random tokens; only hashes are stored. |
| Session tokens | API request runtime | Login flow | Yes | No | Stored as HttpOnly cookies; only token hashes are stored in MySQL. |

| Upper-agent TLS server private key | Upper-computer runtime | Factory OT certificate store / restricted local file | Yes | No | Never committed or copied into the executable; path exists only in the local ignored `agent-config.ini`. |

## Removed From Source

- Hardcoded backend `DATABASE_URL` default.
- Hardcoded bootstrap API key.
- Hardcoded bootstrap administrator password.
- Prefilled login username and password.
- Runnable built-in business data loader.
- Docker/Compose startup path that loaded built-in business data.
- Frontend fallback dashboard with built-in factory, vehicle, point, prediction, and recommendation data.
- Test literals that looked like static API keys or passwords; tests now generate runtime random values.
- Shared frontend server `API_KEY`; user requests now rely only on HttpOnly sessions.
- Pip index build arguments; authenticated `pip.conf` is consumed only through a BuildKit secret mount.

## Enforcement Rules

- `.env`, `.env.*`, `.env.local`, generated logs, local databases, and runtime folders are Git ignored.
- Built-in business data loaders are disabled. Business data must enter through governed import flows or DBA-approved SQL.
- Docker images must start without running DDL, migrations, or seed scripts.
- Any new secret-like input must be added to this inventory before code review.
- `remote_station_connection` may store only certificate/private-key environment reference names, never file contents, private-key material, passwords, tokens, or connection credentials.
- `tools/pq-upper-agent/agent-config.ini`, certificates, generated `.exe`, staged packages, readback files, and agent logs are runtime artifacts and must not be committed.
