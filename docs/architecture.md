# PQ-AI 架构摘要

## 领域边界

- 3C2B 在本项目中表示中涂、色漆、清漆三个涂层体系和工厂定义的两次烘烤。
- 系统记录五个喷涂执行阶段：中涂外喷、色漆一/二遍、清漆一/二遍。
- 质量目标只包括膜厚、色差/效应和橘皮。
- 排除预处理、电泳、涂胶、喷房温湿度、烘房温度、调漆间和光泽度。

## 核心数据链

```text
生产事件
  -> 三个涂层体系 / 五个执行阶段实绩
  -> Dürr 机器人 / 旋杯 / 控制器 / 程序与轨迹版本
  -> 刷子或轨迹段参数与实际执行值
  -> 材料批次与受治理的材料特性
  -> 按目标族版本化的刷子/轨迹到测量点贡献
  -> 点位工艺特征
  -> BYK/Fischer 仪器、方法、校准与原始测量
  -> 橘皮 / 色差效应 / 膜厚结果
  -> 预测 / 诊断 / 推荐
  -> 受控试验 / 审批 / 执行 / 复测 / 持续效果评价
```

## 边界

- MySQL 是业务事实和审计记录的唯一主数据库。
- 当前不可变训练数据集快照、切分成员、目标测量血缘、评估和验收决策保存在 MySQL；正式原始文件和模型制品应存放在对象存储，MySQL 保存其元数据与血缘。
- 工艺推荐必须经过硬安全边界检查和人工审批，不允许首版自动修改机器人参数。
- 配置值和生产实绩值必须分开存储，AI 训练优先使用实绩值。
- 通用 JSON 只允许作为接入缓冲；进入正式特征前必须经过字段定义、单位、方法、来源和范围治理。
- 测量值必须可追溯到仪器型号/序列号、固件/导入模板、探头/方法、校准/参考、重复测量和操作者。
- 训练、验证和测试必须按生产事件/车身/批次分组并按时间留出，禁止随机拆分同一生产事件的点位行。
- 候选模型必须生成时间留出、生产分组留一和可用工厂/车型/颜色验证轴证据；数据缺少轴多样性时必须显式记录。
- 候选模型必须登记覆盖模型载荷、评估指标、数据集引用和训练样本数的工件哈希，验收和激活时必须复核一致。
- 模型解释默认只表示关联；因果诊断和推荐必须有 DOE 或受控变更证据。
- 同一车身/生产运行的点位行不得跨训练和验证集；候选模型只在训练切分拟合，只有独立验证和人工验收通过后才能激活。
- 生效点位特征使用 `point-features-v4-material-governed`，按目标族保存程序版本、贡献版本、轨迹程序、生产设备执行、材料批次结果和材料规格血缘；实际轨迹校验和不匹配或必需材料结果未在投产前验证时阻断特征生成。
