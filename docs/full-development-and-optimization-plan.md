# PQ-AI 完整开发与优化计划

## 1. 当前状态与结论

当前版本已经具备可验证的业务主链、AI 闭环与集成任务框架，可用于本地体验和现场数据接入准备；正式工厂生产上线仍需完成真实系统凭证、历史数据验证、企业 SSO、备份灾备和模型验收。

已具备：

- Next.js 管理端、FastAPI 后端、MySQL/Alembic 数据库结构。
- 工厂、车型、颜色、零件、测量点、程序、刷子、材料、生产、质量和 AI 核心领域模型。
- 42 个工艺/材料参数目录、67 个质量指标目录。
- 点位贡献权重聚合、质量标准判定、基础模型训练、预测、诊断、推荐、审批、执行和复测评价。
- API Key、RBAC、写操作审计和审计中心。

已确认缺口：

- 真实 MES、QMS、机器人/PLC 和材料系统的地址、字段映射与凭证尚未提供。
- 模型漂移监控和正式模型制品存储仍待完成。
- 企业 SSO、备份、灾备、告警、模型漂移监控和真实工厂数据验证仍待完成。

## 2. 目标架构

```text
浏览器 / Next.js 管理端
  -> FastAPI 业务 API / RBAC / 审计
  -> MySQL 业务与模型元数据
  -> 异步集成与 AI Worker
  -> MES / QMS / 机器人 PLC / 材料与测量设备
```

所有质量数据和工艺实绩最终通过“生产事件 + 测量点”关联。所有 AI 推荐必须经过约束校验、人工审批、执行记录和复测评价。

## 3. 分阶段开发步骤

### 阶段 A：可用基础与真实数据库

1. 连接本机 MySQL，创建 `pq_ai` 数据库并执行全部 Alembic 迁移。
2. 初始化幂等演示数据和管理员角色，验证 API、前端、数据库三端连通。
3. 全站切换为浅色主题，完成桌面端和移动端视觉回归。
4. 增加统一错误提示、成功提示、加载状态、空状态和确认对话框。

验收：页面显示 API 实时数据；数据库可查到真实记录；无无响应按钮。

### 阶段 B：主数据完整 CRUD

1. 工厂、车型、颜色、零件、测量编组、测量点完整增删改查。
2. 工厂-车型、车型-颜色、编组-点位关系维护。
3. 颜色 TDS/MSDS/COA/DOE 文件元数据和数字标准维护。
4. 搜索、筛选、分页、导出、字段校验、重复编码检查和停用机制。

验收：用户可从界面建立完整工厂业务上下文，所有操作写入 MySQL 并进入审计。

### 阶段 C：程序、刷子与贡献权重

1. 喷涂程序、版本、适用车型和颜色完整 CRUD。
2. 刷子表、刷子号和参数矩阵编辑，支持批量复制和版本差异。
3. 刷子到测量点贡献权重编辑、归一化校验和审批。
4. 封样、版本审批、生效、退役和回滚流程。

验收：可从界面配置五工序程序，并追溯每个测量点的刷子贡献。

### 阶段 D：生产、材料、质量与标准

1. 生产事件、五工序阶段实绩、实际参数和材料批次维护。
2. 橘皮、色差、光泽度、膜厚数据导入、校验、查询和导出。
3. 质量标准 CRUD、适用范围匹配、自动判定和异常复核。
4. SPC、趋势、车型颜色对比、点位热力图和数据质量监控。

验收：任一质量测量均可追溯到生产事件、程序版本、材料批次和点位特征。

### 阶段 E：AI 工作台与模型治理

1. 训练数据选择、特征版本、模型训练、评估、注册、激活和退役。
2. 预测任务、风险列表、诊断证据和推荐约束可视化。
3. 推荐审批、执行、新程序版本、复测和效果评价操作台。
4. 正式模型、SHAP/敏感性分析、漂移监控、降级和再训练策略。

验收：完整闭环可由用户界面执行，每一步均可解释、可审批、可追溯。

### 阶段 F：生产集成与运维

1. MES、QMS、机器人/PLC、材料和测量设备适配器。
2. 幂等集成事件、失败重试、死信、重放、数据映射和监控。
3. 企业 SSO、密钥托管、字段级权限、备份、灾备和告警。
4. Docker/内网部署、Vercel 前端部署、CI/CD、可观测性和灰度发布。

验收：以工厂为单位完成真实数据试运行，并达到约定的数据完整率、模型精度和闭环收益。

## 4. 横向质量要求

- 所有写操作必须鉴权并进入审计。
- 主数据优先停用而非物理删除；存在引用的资源禁止删除。
- 所有列表支持搜索、筛选、分页、导出和空状态。
- 所有表单具备前后端双重校验、错误反馈和重复提交保护。
- 所有模型结果保存模型版本、特征快照、输入上下文和执行结果。
- 每阶段必须执行 API 测试、MySQL 迁移回滚、前端构建和浏览器端到端回归。

## 5. 当前执行顺序

当前从阶段 A 开始，随后依次完成 B、C、D、E、F，不再将占位交互视为已完成功能。
