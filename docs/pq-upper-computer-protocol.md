# PQRP/1 上位机参数交换协议与使用说明

## 1. 目的与安全边界

PQRP/1（PQ-AI Remote Parameter Protocol）用于 PQ-AI 后端与目标上位机上的轻量代理交换“完整喷涂刷子表”。它运行在 TCP/IP 上，但强制使用双向 TLS，不支持匿名连接和明文通信。

系统不会直接使用未经工厂或 Dürr 批准的机器人写入协议：

- `SIMULATOR`：只修改代理本机模拟数据，用于开发、培训和联调。
- `FILE_DROP`：把批准后的参数包原子写入厂家认可的导入目录，等待上位机已有导入程序处理。
- `DURR_APPROVED_ADAPTER`：调用工厂/Dürr 批准并签名的本地适配器程序。

在取得目标工厂的 DXQ/PLC/上位机接口说明并完成 FAT/SAT 之前，生产环境只能使用只读回传或 `FILE_DROP`，不得把自定义协议直接接到机器人控制器。

## 2. 数据隔离保证

一次参数调整按以下状态流转：

1. `DRAFT`：云端生成完整新版本；远程端无变化。
2. `SUBMITTED`：提交审批；远程端无变化。
3. `APPROVED`：不同人员审批通过；远程端无变化。
4. `STAGED`：完整刷子表传入代理暂存目录；生产参数无变化。
5. `LOCAL_CONFIRMED`：上位机现场人员在本机页面确认；生产参数仍无变化。
6. `WAITING_READBACK`：参数包已经交给厂家导入目录或批准适配器，但上位机尚未生成一致回读；系统禁止重复下发，只允许重新核验。
7. `APPLIED`：代理已完成可同步验证的批准适配方式；开始影响目标参数。
8. `VERIFIED`：重新只读回传完整刷子表，哈希与候选版本一致。

现场人员也可以在本机页面拒绝暂存包。拒绝结果回传后发布单进入 `REJECTED`，不能继续提交。

后端只允许上述顺序前进。申请人不能审批自己的发布单；没有云端审批证据不能暂存；没有本机确认标记不能提交；回读不一致不能标记成功。

## 3. 传输格式

每条消息是：

```text
4 字节无符号大端长度 + UTF-8 消息正文
```

长度必须大于 0 且不超过连接配置的 `maxPackageBytes`。连接一次只处理一个请求，然后关闭，避免长连接状态污染。

消息正文包含：

| 字段 | 说明 |
|---|---|
| `protocol` | 固定 `PQRP/1` |
| `messageId` | 本次消息唯一编号 |
| `replyTo` | 响应对应的请求编号 |
| `sentAt` | UTC 时间；代理仅接受与本机时间相差 5 分钟以内的请求 |
| `nonce` | 一次性随机数；10 分钟内禁止重复 |
| `agentId` | 目标代理编号，必须与配置一致 |
| `type` | 消息类型 |
| `body` | 消息业务内容 |
| `bodyHash` | 对业务内容做确定性排序后计算的 SHA-256 |

身份认证、机密性和传输完整性由双向 TLS 证书提供；`bodyHash` 用于参数包落盘、回读和跨系统对账，不替代 TLS 身份认证。

## 4. 消息类型

| 请求 | 响应 | 是否改变生产参数 | 用途 |
|---|---|---:|---|
| `HELLO` | `HELLO_ACK` | 否 | 验证证书、代理编号、版本和适配方式 |
| `INVENTORY_REQUEST` | `INVENTORY_RESPONSE` | 否 | 只读拉取上位机最新完整刷子表 |
| `PREPARE_RELEASE` | `PREPARE_ACK` | 否 | 校验审批证据和哈希，仅写暂存目录 |
| `RELEASE_STATUS_REQUEST` | `RELEASE_STATUS_RESPONSE` | 否 | 查询现场人员是否确认或拒绝 |
| `COMMIT_RELEASE` | `COMMIT_ACK` | 是 | 现场确认后调用批准的适配方式 |
| 任意非法请求 | `ERROR` | 否 | 返回通俗错误原因 |

## 5. 完整刷子表

每次发布都传输完整版本，而不是差量命令。完整版本至少包括：

- 程序编号、程序名称、工段、工作站、版本号；
- 该版本全部刷子表和全部刷子号；
- 每个刷子号的喷涂流量、旋杯转速、外成型空气、内成型空气、静电高压及已治理的其他参数；
- 参数单位；
- 完整表 SHA-256。

为避免 Python、JavaScript 和厂家文件对 `320.0`、`320`、科学计数法产生不同哈希，协议中的参数 `value` 固定使用规范十进制字符串：无前导正号、无多余前导或末尾零、不使用科学计数法，例如 `"320"`、`"0.125"`。后端生成发布包时负责转换；通讯程序拒绝不规范的发布值，并在厂家回读后先规范化再计算哈希。

云端“改一个值”时，后端也先复制原版本全部刷子和参数，再应用这一个修改并生成新版本。因此远程端永远收到可独立核验和回退的完整版本。

## 6. 三方同步与差异处理

系统分别保存以下不可变快照：

- `CLOUD`：云数据库中的完整程序版本；
- `VIRTUAL_LINE`：虚拟生产线当前采用的完整版本；
- `UPPER_COMPUTER`：通过 `INVENTORY_REQUEST` 回传的真实上位机版本。

对账以“刷子表/刷子号/参数”为键，显示三方值、一致、不同或缺少。对账本身只读。修改云端或虚拟线必须生成新版本；修改真实上位机必须重新走完整发布审批，不能在差异表中直接覆盖。

## 7. 证书和网络配置

- 工厂 PKI 分别签发后端客户端证书和上位机代理服务端证书。
- 后端数据库只保存运行环境变量名，例如“客户端证书配置名”，不保存证书、私钥路径或内容。
- 证书文件路径在 API 容器/主机启动时通过环境变量注入。
- 代理证书路径只保存在上位机本地 `agent-config.ini`，私钥目录只授予代理服务账号读取权限。
- 防火墙只允许指定 API 主机访问代理 TLS 端口；本机确认页面固定绑定 `127.0.0.1`，不能从网络访问。
- 上位机和 API 主机必须使用受治理的时间同步源，否则五分钟防重放窗口会拒绝请求。

双向 TLS 与证书信任列表的设计与 OPC UA 的应用证书、信任列表、签名和加密原则一致，但 PQRP/1 不是 OPC UA，也不声明与任何 Dürr 私有接口兼容。

## 8. 部署与操作

1. 在隔离的 Windows 构建机执行 `tools/pq-upper-agent/build-windows.ps1`。
2. 对 `.exe` 做公司代码签名、病毒扫描和 SHA-256 登记。
3. 通过批准的离线介质把 `.exe`、`agent-config.ini`、证书和批准适配器送入目标上位机。
4. 先使用 `SIMULATOR` 做证书、暂存、现场确认、提交和回读全流程联调。
5. 通过 FAT/SAT 后，切换为 `FILE_DROP` 或 `DURR_APPROVED_ADAPTER` 并重新审批连接。
6. 在 PQ-AI “系统设置 → 机器人远程工作站”填写相同代理编号、地址、端口和证书配置名。
7. 点击“测试只读连接”；测试只执行 `HELLO`，不会读取或修改参数。
8. 分别登记云端、虚拟线并只读拉取上位机快照，先完成三方对账。
9. 参数发布必须由申请人、审批人和现场确认人按职责完成。

## 9. 故障与回退

- 网络中断：发布单保持原状态，代理不应用不完整消息；可安全重试同一哈希的暂存请求。
- 同一发布单号内容不同：代理拒绝，防止覆盖已审核包。
- 现场未确认：代理拒绝提交。
- 厂家导入未完成：`FILE_DROP` 返回“已交付、等待回读”，不报告成功，也不允许重复下发；用户只能点击“重新读取并核对”。
- 回读不一致：云端标记失败，保留原版本为回退版本；不得自动再次写入。
- 需要回退：以原版本重新生成一张发布单并重复审批，不使用隐藏的“强制回滚”命令。

## 10. 技术依据

- Node.js Single Executable Application 可把应用发布到未安装 Node.js 的目标系统；Node.js 25.5 起可直接使用 `--build-sea` 生成可执行文件。
- 工业网络部署应使用分区、最少连接、防火墙白名单、审计和职责分离。
- Dürr/DXQ 的实际参数写入仍以工厂采购版本、批准接口文件和现场验收结果为准。
